# Usage Guidenn## Quick Startnn1. **Setup your project** (see [Setup Guide](setup.md))n2. **Edit initial.md** with project detailsn3. **Run the workflow**: `./workflows/enforce-prp-workflow.sh`n4. **Review and customize** the generated PRPn5. **Begin implementation** following the PRPnn## Workflow OverviewnnThe PRP workflow consists of three main steps:nn### Step 1: Read Contextn- Parses `initial.md` for project detailsn- Validates required information is presentn- Extracts language, purpose, and requirementsnn### Step 2: Update Templaten- Detects project language automaticallyn- Updates base PRP template with language-specific patternsn- Creates versioned templates for reusenn### Step 3: Create PRPn- Generates project-specific PRP documentn- Replaces placeholders with actual contentn- Creates backups and metadatann## CLI Commandsnn### prp creatennCreates a new PRP based on project context:nn```bashnprp create                    # Create with default settingsnprp create --template=nextjs  # Use Next.js specific templatenprp create --output=my-prp.md # Specify output filenamen```nn### prp executennExecutes the implementation plan from a PRP:nn```bashnprp execute                    # Execute entire PRPnprp execute --phase=1          # Execute specific phasenprp execute --task="Setup DB"  # Execute specific tasknprp execute --dry-run          # Show execution plann```nn### prp validatennValidates a PRP against quality standards:nn```bashnprp validate                    # Validate current PRPnprp validate --fix              # Auto-fix common issuesnprp validate --strict           # Use strict validation rulesn```nn## Customizationnn### Modifying TemplatesnnTemplates are located in `templates/` directory:nn- `prp-base.md` - Base template for all projectsn- `language-hints/` - Language-specific patternsn- `versioned-templates/` - Saved language versionsnn### Adding Language Supportnn1. Create new language hint file: `templates/language-hints/yourlang.md`n2. Follow the existing pattern structuren3. Include common patterns, project structure, and examplesn4. Test with a sample projectnn### Custom Validation RulesnnCreate custom validation rules in JSON format:nn```jsonn{n  "company_standards": {n    "required_sections": ["security", "compliance"],n    "quality_threshold": 0.9n  },n  "architecture_guidelines": {n    "preferred_patterns": ["microservices", "event-driven"],n    "forbidden_patterns": ["monolith", "tight-coupling"]n  }n}n```nn## Implementationnn### Following the PRPnn1. **Review the generated PRP** thoroughlyn2. **Understand the phases** and their dependenciesn3. **Start with Phase 1** (Foundation)n4. **Complete each task** before moving to the nextn5. **Update progress** in the PRP as you workn6. **Run validation** regularly to ensure qualitynn### Task Managementnn- Each task should be **atomic** (1-4 hours of work)n- **Dependencies** should be clearly identifiedn- **Success criteria** must be measurablen- **Error handling** should be plannednn### Quality Assurancenn- **Code reviews** for each completed taskn- **Testing** at multiple levels (unit, integration, e2e)n- **Documentation** updates as implementation progressesn- **Performance** monitoring and optimizationnn## Troubleshootingnn### Common
